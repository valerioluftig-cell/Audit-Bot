Technical Overview

Louisiana Parish Audit Pipeline

An automated pipeline that reads scanned parish audit PDFs, detects four standard financial statements using keyword pattern matching, parses line items via Claude, and writes the results to structured Excel workbooks for cross-parish analysis.

64 Parishes Fiscal Year 2013 4 Statement Types Python 3.11 Anthropic Claude FastAPI
Architecture
Five-stage pipeline

Each parish audit PDF passes through five sequential stages. Text extraction and page detection run locally with no API calls; the Claude API is invoked once per parish with the assembled statement text, and results are cached to disk so re-runs are free.

Stage 2
Page detection — how it works

The core challenge is that PDF text extraction from scanned documents is unreliable. Rather than looking for exact strings, the detector uses layered logic: OCR-variant keyword patterns that must all match, exclude lists that reject lookalike pages, a notes-page guard that inspects only the title area, and a confirmation window that verifies data content before committing to an anchor page.

Notes-page guard — checks title area only, not footers pipeline/extractor.py 
def _is_notes_page(text: str) -> bool:
    # Notes title is always in the first ~200 chars (parish + title + date).
    # Checking only the top means a "see notes to financial statements" footer
    # on a CBS page does NOT fire this guard and stop collection early.
    first400 = text[:400]
    very_top = text[:200]
    return (
        "notes to financial"   in first400
        or "notes to the financial" in first400
        or "notes to the basic"    in first400
        or ("notes to the" in very_top and "statements" in very_top)
        or ("notes to"     in text[:150] and "statements" in text[:150])
    )

Core pattern matcher — applied to every page for every statement type pipeline/extractor.py 
def _matches_any(text: str, patterns: dict, allow_notes: bool = False) -> bool:
    # Notes pages: title check first so CBS/SOA footer refs don't fire
    if not allow_notes and _is_notes_page(text):
        return False

    # Table of contents pages list every statement title but contain no data.
    # Dotted leaders (10+ periods) uniquely identify TOC formatting.
    if "table of contents" in text[:400] or "........." in text:
        return False

    # All exclude terms must be absent
    for excl in patterns.get("exclude", []):
        if excl in text:
            return False

    # Any one require_any group must fully match (AND within group, OR across groups)
    for req_group in patterns.get("require_any", []):
        if all(kw in text for kw in req_group):
            return True
    return False
Confirmation window — prevents false positives from TOC or intro pages pipeline/extractor.py · find_statement_pages() 
anchor = None
for i, text in enumerate(texts):
    if _matches_any(text, patterns):
        # Require at least one confirmation keyword within ±3 pages.
        # A TOC page lists statement names but won't have "fund balance", etc.
        window = " ".join(texts[max(0, i-1):min(n, i+4)])
        if any(kw in window for kw in CONFIRMATION[stmt_type]):
            anchor = i
            break

CONFIRMATION = {
    "cbs":  ["assets", "liabilities", "fund balance"],
    "soa":  ["expenses", "general revenues"],
    "sona": ["primary", "assets", "liabilities"],
    "ca":   ["depreciation", "beginning balance", "increases"],
}

Engineering challenge
OCR corruption — real examples

Louisiana parish audits are scanned documents. OCR produces systematic corruptions — letter substitutions, word merges, inserted punctuation — that differ by scanning hardware and scanner settings. The pipeline handles these with layered pattern variants rather than single exact-match strings.

Why this matters: Management Discussion & Analysis pages describe capital assets and financial activities in narrative form. Their text overlaps heavily with the actual financial statements — the same keywords appear, but in sentence context rather than table headers. Without a dedicated MDA_EXCLUDE list and the notes-page guard, these pages would anchor the detector and the wrong pages would be sent to Claude.

Output
The four financial statements

Every Louisiana parish audit contains these four statements. The pipeline produces a separate Excel workbook for each, with one worksheet per parish, enabling cross-sectional comparison across all 64 parishes.

CBS
Combined Balance Sheet
Governmental funds view. Each major fund (General, Debt Service, Capital Projects, etc.) is a column. Shows what the government holds in cash and receivables, what it owes in payables and deferred inflows, and the remaining fund balance — broken down by spendability classification (nonspendable, restricted, committed, assigned, unassigned). The most structurally variable statement across parishes.
Cash & deposits Taxes receivable Accounts payable Deferred inflows Fund balances by type
SOA
Statement of Activities
Government-wide income statement equivalent. Each function (public safety, public works, health, culture, etc.) shows total expenses, program revenues earned back (charges for services, operating grants, capital grants), and the net cost covered by general revenues. Property taxes and sales taxes appear here, along with change in net position.
Expenses by function Charges for services Operating & capital grants Property taxes Sales taxes Change in net position
SONA
Statement of Net Position
Government-wide balance sheet. Unlike the CBS, this includes long-lived capital assets (infrastructure, buildings, equipment) and long-term liabilities (bonds, pension obligations, compensated absences). Net position = total assets + deferred outflows − total liabilities − deferred inflows. Broken into invested in capital assets, restricted, and unrestricted.
Capital assets net Long-term liabilities Deferred outflows Deferred inflows Net position components
CA
Capital Assets Schedule
Tracks every asset category from beginning to ending balance across the fiscal year: land (not depreciated), construction in progress, buildings, machinery, infrastructure. Shows additions (new purchases or completions), disposals, and annual depreciation charges. May appear as a primary financial statement or as Note 4 / Note 5 inside the notes section.
Land Infrastructure Buildings Machinery Additions / Disposals Accumulated depreciation
Stage 1 & 5
Web interface — upload and live streaming

A FastAPI server accepts batch PDF uploads, runs the pipeline in a background thread, and streams each log line to the browser via Server-Sent Events. The user sees a live-scrolling terminal log and download buttons appear when all four workbooks are ready.

Background job + SSE stream — browser receives each log line in real time webapp/app.py 
@app.post("/run")
async def run(files: list[UploadFile] = File(...), year: int = Form(...)):
    job_id = str(uuid.uuid4())[:8]
    q: queue.Queue[str] = queue.Queue()
    JOBS[job_id] = {"queue": q, "status": "running", "output_dir": str(out_dir)}

    def _run():
        run_pipeline(..., progress_callback=lambda msg: q.put(msg))
        q.put("__DONE__")   # sentinel — closes the SSE stream

    threading.Thread(target=_run, daemon=True).start()
    return {"job_id": job_id}


@app.get("/progress/{job_id}")
def progress(job_id: str):
    def stream():
        q = JOBS[job_id]["queue"]
        while True:
            try:
                msg = q.get(timeout=25)
            except queue.Empty:
                yield "data: \n\n"   # heartbeat — keeps connection alive
                continue
            yield f"data: {json.dumps(msg)}\n\n"
            if msg == "__DONE__": break

    return StreamingResponse(
        stream(),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"},
    )
